MCP Server 入門模板：快速開始

每一步都有：要打的指令 → 真實輸出 → 成功會看到什麼。不需要任何 API key，這個 starter 只用 Python 標準函式庫。

前置需求

  • Python 3.10+
  • Git，會用終端機
  • uv（本教學的環境管理工具）

安裝 uv（一次就好）

Ubuntu / macOS：curl -LsSf https://astral.sh/uv/install.sh | sh
Windows（PowerShell）：powershell -ExecutionPolicy ByPass -c "irm https://astral.sh/uv/install.ps1 | iex"
裝完重開終端機，uv --version 印得出版本就 OK。uv sync 會依 pyproject.toml + uv.lock 自動建立 .venv 並裝好套件，uv run 直接在那個環境執行。以下 uv sync / uv run 在 Ubuntu 與 Windows 完全相同，平台差異只在安裝 uv 這一步。

步驟 1：取得程式

git clone https://github.com/yazelin/mcp-server-starter.git
cd mcp-server-starter
uv sync

uv sync 的真實輸出（會建立 .venv 並 build 本專案）：

Using CPython 3.11.13
Creating virtual environment at: .venv
Resolved 1 package in 1ms
   Building mcp-server-starter @ file:///path/to/mcp-server-starter
      Built mcp-server-starter @ file:///path/to/mcp-server-starter
Prepared 1 package in 942ms
Installed 1 package in 0.76ms
 + mcp-server-starter==0.1.0 (from file:///path/to/mcp-server-starter)

步驟 2：跑 smoke test

它會用 subprocess 啟動 server.py，依序送 initialize、tools/list、tools/call。

uv run python client_smoke_test.py

真實輸出（實際跑出來貼上）：

{"jsonrpc": "2.0", "id": 1, "result": {"protocolVersion": "2024-11-05", "capabilities": {"tools": {}}, "serverInfo": {"name": "mcp-server-starter", "version": "0.1.0"}}}
{"jsonrpc": "2.0", "id": 2, "result": {"tools": [{"name": "echo", ...}, {"name": "now", ...}, {"name": "read_text_file", ...}]}}
{"jsonrpc": "2.0", "id": 3, "result": {"content": [{"type": "text", "text": "hello"}], "isError": false}}

成功會看到：三行 JSON，id 1/2/3 都有，id:3 的 echo 回 "hello"、isError 為 false。

步驟 3：手動把 JSON-RPC 餵進 server

printf '%s\n' \
'{"jsonrpc":"2.0","id":3,"method":"tools/call","params":{"name":"now","arguments":{}}}' \
| MCP_WORKSPACE="$PWD" uv run python server.py

真實輸出（時間會跟著你目前時間變）：

{"jsonrpc": "2.0", "id": 3, "result": {"content": [{"type": "text", "text": "2026-06-02T03:28:51+08:00"}], "isError": false}}

第一次成功的標準

  • uv run python client_smoke_test.py 印出三行 JSON，isError 為 false。
  • 手動 pipe 時 now 回了一個合理時間。
  • read_text_file 能讀到 MCP_WORKSPACE 內的檔案。
  • 沒有把 secret 或本機絕對路徑誤 commit 到 GitHub。

MCP Server 入門模板：架構說明

核心檔案

  • server.py：單檔 MCP JSON-RPC stdio server
  • tools()：宣告 echo / now / read_text_file / word_count 工具（word_count 是 03 教學示範新增的範例工具）
  • call()：依工具名稱執行實際邏輯
  • safe()：用 MCP_WORKSPACE 做最基本的檔案安全邊界
  • client_smoke_test.py：用 subprocess 模擬 MCP client 呼叫 server

資料流

這是純 stdio 的 MCP server，沒有 HTTP framework（不是 FastAPI），傳輸就是標準輸入／輸出上的換行分隔 JSON-RPC。

  1. client（例如 Claude Desktop）以 subprocess 啟動 server，透過 stdin 送進一行 JSON-RPC 請求。
  2. main() 逐行讀取 stdin，json.loads 解析後交給 handle()。
  3. handle() 依 method 分派：initialize / tools/list / tools/call / notifications/initialized。
  4. tools/call 進入 call() 執行對應工具；檔案類工具一律經過 safe() 限制在 MCP_WORKSPACE 內。
  5. 結果以換行分隔 JSON-RPC 寫回 stdout；notification 類訊息不回應。

設計原則

  • 先讓流程可跑，再做漂亮抽象。
  • token 與 secrets 全部放在環境變數。
  • 每一層保持可以替換：入口、AI provider、資料來源、部署方式。
  • 範例程式刻意保持小，方便你看懂後改成自己的版本。

MCP Server 入門模板：帶你走一遍

這整支 server 就是「讀一行 JSON-RPC 請求 → 回一行 JSON-RPC 回應」。核心三個函式：tools()（宣告工具與 schema）、call()（執行邏輯）、handle()（依 method 分派）。所有輸出皆為實際跑出來貼上。

initialize → tools/list → tools/call

tools/call 呼叫 now 的真實回應：

{"jsonrpc": "2.0", "id": 3, "result": {"content": [{"type": "text", "text": "2026-06-02T03:28:51+08:00"}], "isError": false}}

注意回應形狀：result.content 是陣列，每個元素有 type 與 text，isError 表示工具是否成功。

安全邊界：親眼看它擋下逃逸

read_text_file 只能讀 MCP_WORKSPACE 底下的檔案。safe() 用 .resolve() 把 .. / symlink / 絕對路徑攤平成真實路徑再比對。試著逃出 workspace：

printf '%s\n' \
'{"jsonrpc":"2.0","id":5,"method":"tools/call","params":{"name":"read_text_file","arguments":{"path":"../sibling-secret.txt"}}}' \
'{"jsonrpc":"2.0","id":6,"method":"tools/call","params":{"name":"read_text_file","arguments":{"path":"/etc/passwd"}}}' \
| MCP_WORKSPACE="$PWD" uv run python server.py

真實回應（兩個都被擋）：

{"jsonrpc": "2.0", "id": 5, "result": {"content": [{"type": "text", "text": "Path escapes workspace"}], "isError": true}}
{"jsonrpc": "2.0", "id": 6, "result": {"content": [{"type": "text", "text": "Path escapes workspace"}], "isError": true}}

這是這個 starter 最重要的一課：把檔案讀取工具交給 AI 之前，邊界要先擋死。它回的是 isError:true 的乾淨工具錯誤，而不是 transport 錯誤。

加一個你自己的工具（兩個地方）

加工具永遠改兩處：tools() 加 schema、call() 加邏輯。範例 word_count 數一段文字有幾個詞。

tools() 末尾新增一筆 word_count schema；call() 加一行：

if name=="word_count": return str(len(args.get("text","").split()))

呼叫的真實回應（"the quick brown fox jumps" 共 5 詞）：

{"jsonrpc": "2.0", "id": 2, "result": {"content": [{"type": "text", "text": "5"}], "isError": false}}

動手練習

換你加一個 reverse 工具：tools() 仿照 echo 加一筆，call() 加 if name=="reverse": return args.get("text","")[::-1]。呼叫 reverse 傳 "hello" 預期回 "olleh"、isError 為 false。若回 "Unknown tool: reverse"，代表 schema 加了但 call() 忘了加邏輯。

MCP Server 入門模板：部署與接入

這是純 stdio 的 MCP server：沒有 HTTP port、沒有 /health endpoint，也不部署到 Render / Railway / Fly.io。MCP client（例如 Claude Desktop）會把它當 subprocess 啟動，透過標準輸入／輸出交換 JSON-RPC。「部署」其實是「讓 client 知道怎麼啟動這支程式」。

接入前檢查

  • 已跑過 uv sync，本機流程跑通（uv run python client_smoke_test.py 印出三段 JSON-RPC 回應）。
  • 設好 MCP_WORKSPACE，read_text_file 只能讀到預期目錄。
  • server.py（或安裝後的 mcp-server-starter 指令）是 client 能存取的絕對路徑。
  • stdout 只輸出 JSON-RPC，debug 訊息寫 stderr，否則破壞 framing。

在 Claude Desktop 設定啟動方式

本教學用 uv 管理環境，讓 client 透過 uv run 啟動最穩：command 設 uv，用 --project 指向 repo 絕對路徑，這樣不論 client 從哪個工作目錄啟動都找得到正確的 .venv 與 console script。此形式已實測過協定正常、MCP_WORKSPACE 仍由 env 獨立決定：

{
  "mcpServers": {
    "mcp-server-starter": {
      "command": "uv",
      "args": ["run", "--project", "/absolute/path/to/mcp-server-starter", "mcp-server-starter"],
      "env": { "MCP_WORKSPACE": "/absolute/path/to/your/workspace" }
    }
  }
}

若 uv 不在 client 的 PATH 上，把 command 換成 uv 絕對路徑（Ubuntu/macOS 通常是 ~/.local/bin/uv，Windows 通常是 %USERPROFILE%\.local\bin\uv.exe）。設定後重啟 client，server 會在需要時被自動啟動，用完即關，不需常駐。

想放到雲端？

若要讓遠端 client 透過網路連線，需自行加上 HTTP / SSE transport。本 starter 只示範 stdio，不包含 HTTP/SSE transport。

實務提醒

接入不是最後一步。正式使用前至少要補：log（寫 stderr，別污染 stdout）、錯誤告警、workspace 權限控管、備份策略，以及對 read_text_file 之類工具的存取範圍稽核。

MCP Server 入門模板：常見踩雷清單

1. print 到 stdout 污染 JSON-RPC

stdio MCP server 的 stdout 只能放 JSON-RPC。只要 print 一個 debug 訊息，client 解析第一行就炸：

json.decoder.JSONDecodeError: Expecting value: line 1 column 1 (char 0)

修法：debug / log 一律寫 stderr，例如 print("...", file=sys.stderr)。

2. method 名稱寫錯（listTools 不是 tools/list）

method 是固定字串且有斜線。送 listTools 的真實回應：

{"jsonrpc": "2.0", "id": 7, "error": {"code": -32601, "message": "Method not found: listTools"}}

-32601 是 JSON-RPC 標準的「Method not found」。用 tools/list 而不是 listTools。

3. 忘了設 MCP_WORKSPACE

沒設時預設是 server 啟動當下的工作目錄，常常不是你以為的地方，導致「該讀到的讀不到」。請在 client 設定檔 env 明確指定絕對路徑。

4. 加了 schema 卻忘了在 call() 加邏輯

tools/list 看得到工具，但呼叫落到 call() 結尾：

{"jsonrpc": "2.0", "id": 2, "result": {"content": [{"type": "text", "text": "Unknown tool: reverse"}], "isError": true}}

5. 工具缺必填參數

呼叫 read_text_file 沒給 path 的真實回應（好的錯誤：isError:true 加人看得懂的訊息）：

{"jsonrpc": "2.0", "id": 1, "result": {"content": [{"type": "text", "text": "Missing required argument: path"}], "isError": true}}

6. 沒裝 uv，或忘了先 uv sync

本教學用 uv 管理環境。打 uv ... 出現 command not found 代表沒裝 uv（依前置需求安裝後重開終端機，uv --version 印得出版本再繼續）。裝了卻直接 uv run mcp-server-starter 找不到 console script，通常是忘了先在 repo 根目錄跑一次 uv sync（它會建立 .venv 並安裝本專案）。uv sync / uv run 兩平台完全相同，平台差異只在安裝 uv 這一步。

Debug 順序（針對 stdio MCP）

  1. 先用 uv run python client_smoke_test.py 確認協定正常。
  2. 手動 pipe 單一 method 進 uv run python server.py，縮小出錯的步驟。
  3. 確認 stdout 沒被 print 污染（debug 全走 stderr）。
  4. 確認 method 名稱、工具 name 逐字正確。
  5. 確認 MCP_WORKSPACE 指向正確絕對路徑。
  6. 看完整錯誤與 isError 欄位，別只看最後一行。

MCP Server 入門模板：改成你的使用場景

可延伸方向

  • 把內部 REST API 包成工具：例如查訂單、查庫存、查機台狀態。
  • 加入 whitelist：只允許讀特定資料夾或呼叫特定 endpoint。
  • 加上 audit log：記錄每次工具呼叫參數與結果。
  • 升級到 FastMCP：當你理解底層協定後再使用框架。

改造原則

  • 一次只改一個層次：先改資料來源，再改 UI，再改部署。
  • 先保留原本可跑的範例，另開 branch 做實驗。
  • 每加一個外部服務，就加一個健康檢查或 smoke test。
  • 先做 PoC，再決定要不要產品化。

適合拿來做課程 / 工作坊的題目

  • 從零跑起這個 starter。
  • 改成自己的真實場景。
  • 加入權限、部署與監控。
  • 現場 debug 學員遇到的問題。
